1 · Iterable → Iterator → Generator

可迭代对象 → 迭代器 → 生成器

These three are not three rivals — they're three layers of the same idea. The trick is to ask one question about any object: "What does it promise?"

📚

Iterable

has __iter__

"I can hand you a fresh reader." A book on the shelf. Re-usable: ask twice, get two readers.

可迭代对象：有 __iter__，能交给你一个全新的读取器。像书，可反复借阅。

🔖

Iterator

has __next__ (+__iter__)

"I remember where I am; click me for the next item, or StopIteration." A bookmark. One-shot: once read, it's spent.

迭代器：有 __next__，记住当前位置。像书签，一次性，读完即耗尽。

📼

Generator

a function with yield

An iterator you got for free: any function that yields. Python writes __next__ + StopIteration for you.

生成器：带 yield 的函数，本身就是迭代器，Python 替你写好了 __next__。

🔁 RECALL B3 — two ways to make for x in v: work

A for loop doesn't need __iter__ specifically — it needs one of two protocols. Step through each to see the exact calls Python makes.

让 for 能工作有两条路：要么 __iter__（新式协议），要么 __getitem__（旧式回退）。逐步看 Python 实际调用了什么。

__iter__ (generator)
__getitem__ (fallback)
def __iter__(self):
    yield self.x      # 1st next() → 3
    yield self.y      # 2nd next() → 4; function ends → StopIteration
def __getitem__(self, i):
    return (self.x, self.y)[i]   # v[0]=3, v[1]=4, v[2] → IndexError
# Step through `for x in v:`

Both yield 3 then 4. The difference is the end signal: a generator just ends (auto StopIteration); the index protocol walks 0,1,2… until IndexError. Your vector2d.py ships both — that's why list(v), x,y = v, and v[0] all work.

So how do they nest?

三者如何嵌套？

ITERABLE — anything with __iter__ 📚 book · str, list, dict, set, your Vector2D · re-iterable
ITERATOR — also has __next__, and __iter__ returns self 🔖 bookmark · stateful · one-shot
GENERATOR — an iterator the compiler wrote from yield 📼 the easy way to get the middle ring

Every generator is an iterator; every iterator is an iterable (its __iter__ just returns itself). The reverse is false: a list is iterable but not its own iterator — that's why iter(mylist) gives you a separate cursor each time.

每个生成器都是迭代器；每个迭代器都是可迭代对象（其 __iter__ 返回自身）。反之不成立：list 是可迭代的，但不是自己的迭代器 —— 所以每次 iter(list) 都给你一个独立游标。

2 · What __hash__ really does — and why it's necessary

2 · __hash__ 底层做什么，以及为什么必要

A set / dict needs to answer "is this item in here?" in O(1) — without scanning every element. The trick: a hash function turns the object into a number, and that number tells you which bucket to look in. You check one bucket, not the whole collection.

set/dict 要在 O(1) 内回答"这个元素在不在里面？"，不能逐个扫描。诀窍：哈希函数把对象变成一个数字，这个数字直接告诉你去哪个桶找。你只查一个桶，而非整个集合。

A hash function must be deterministic: the same key must always produce the same number, or you'd drop it in one bucket and look in another. (Here hash("cat") = sum of char codes, for show; CPython's real one is fancier but obeys the same rule.)

🎯 This is the bug hiding in your own vector2d.py.

You defined __hash__ as hash((self.x, self.y)) — correct only if the vector is treated as immutable. But x and y are plain settable attributes. Put v in a set, then do v.x = 9, and v becomes a ghost: still sitting in its old bucket, but v in myset now returns False (state ③ above). That's the exact reason __add__ must return a new object instead of mutating (RECALL Part C) — and the deeper answer to your homework Q1, which you'd half-answered as "avoid hash collisions."

你把 __hash__ 写成 hash((x, y)) —— 只有把向量当作不可变时才正确。但 x、y 是可随意修改的属性。把 v 放进 set 后再 v.x = 9，v 就成了"幽灵"：还在旧桶里，但 v in myset 已返回 False。这正是 __add__ 必须返回新对象的原因（Part C），也是作业 Q1 更深层的答案 —— 你当时答的"避免哈希冲突"只对了一半。

The right call for each case

Your object is…Define __hash__?Why
Immutable (Fraction you never mutate, tuple-like)✅ yes — hash((fields))Hash is stable → safe as a dict key / set member.
Mutable (Vector2D with settable x,y)❌ no — leave __hash__ = NoneHash would drift on mutation → ghost entries. Python disables it for you the moment you write __eq__.

So defining __eq__ and getting "unhashable type" is not a bug — it's Python protecting you. You only add __hash__ back when you can promise the object won't change.
